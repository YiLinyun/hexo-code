在 Java Web 应用（尤其是 Spring Boot/Spring MVC）的典型分层架构中，异常处理的职责是分层的，通常建议的做法是：

核心原则：

  1. Service 层： 抛出异常。
  2. Controller 层： 捕获异常并将其转化为对用户友好的响应。
  3. 全局异常处理器： 处理跨越多个 Controller 的通用异常或 Controller 未处理的异常。

📍 详细说明

📍 1. Service 层 - 负责抛出异常

-

职责：

  • 处理核心业务逻辑和数据访问。

  • 在遇到业务规则违反、数据验证失败、数据访问错误、外部服务调用失败等异常情况时，抛出相应的已检查或未检查异常。

抛出的异常类型：

  • 自定义业务异常： 如 UserNotFoundException, InsufficientBalanceException, ProductNotActiveException 等。这些异常应该清楚地表明业务逻辑出错的原因。

  • 已检查异常： 如 IOException, SQLException（但通常推荐将其包装成未检查异常，见下一点）。

  • 未检查异常： 如 IllegalArgumentException, IllegalStateException (用于参数校验失败、非法状态转换等)。

  • 包装异常： 将底层技术异常（如 JDBC 的 SQLException、JPA 的 PersistenceException）包装成自定义的运行时异常（如 DataAccessException 的子类）。Spring 的 DataAccessException 正是这样做的。这样避免了将技术细节泄漏给上层。

  • 为什么在 Service 层抛出？

    • 关注点分离： Service 层的职责是实现业务逻辑。它知道业务操作何时失败以及失败的具体原因（业务规则）。抛出异常是它报告失败状态的标准方式。
    • 事务完整性： Service 层通常声明了事务边界（@Transactional）。在 Service 方法内部抛出异常是触发事务回滚的关键机制。 如果 Service 层内部捕获了异常并悄悄处理掉，可能导致业务操作部分成功但事务没有回滚的数据不一致状态。
    • 避免干扰业务逻辑： 在 Service 层内部写大量 try-catch 来处理异常并返回错误码/信息，会严重污染核心业务代码的清晰度。
    • 向上传播错误： Service 层应该向其调用者（通常是 Controller）报告操作是否成功，以及失败的原因。异常是 Java 中表达失败的强有力机制。

📍 2. Controller 层 - 负责捕获并处理异常

-

职责：

  • 接收用户请求（HTTP 请求）。

  • 调用一个或多个 Service 层方法执行业务逻辑。

  • 捕获 Service 层（或其调用的方法）抛出的异常。

  • 将捕获到的异常转化为对 API 调用者（用户、前端、其他服务）友好的错误响应（如 JSON 格式的错误信息、特定的 HTTP 状态码）。常见的响应格式：{ "status": 400, "error": "Bad Request", "message": "用户ID不能为空" }。

  • 为什么在 Controller 层捕获处理？

    • 关注点分离： Controller 的职责是处理 HTTP 请求/响应。它知道如何将 Java 层的错误对象映射到最终的 API 契约（HTTP 状态码、错误体格式）。

  • 用户友好响应： Service 层抛出的异常（如 SQLIntegrityConstraintViolationException）通常包含技术细节，不应直接暴露给最终用户/调用者。Controller 层负责将这些异常翻译成业务相关的、安全的错误信息（如“用户名已存在”）。

  • 合适的 HTTP 状态码： 不同的业务错误需要映射到不同的 HTTP 状态码（404 资源不存在，400 请求错误，409 冲突，403 禁止访问等）。Controller 层负责选择正确的状态码。

📍 3. 全局异常处理器 (@ControllerAdvice + @ExceptionHandler) - 统一处理跨 Controller 异常

-

职责：

  • 定义一个类，用 @ControllerAdvice (或 @RestControllerAdvice) 注解。

  • 在这个类中，定义多个用 @ExceptionHandler 注解的方法。

  • 这些方法会集中处理应用中所有 Controller 抛出的指定类型的异常。

  • 作用：

  • 避免重复： 将通用异常（如验证失败 MethodArgumentNotValidException，AuthenticationException，未捕获的 RuntimeException，或你的自定义顶层异常 AppBaseException）的处理逻辑写在同一个地方，避免在每个 Controller 写重复的 @ExceptionHandler。

  • 统一响应格式： 确保不同 Controller 抛出的相同异常类型，返回给客户端的错误响应格式是完全一致的。

  • 处理框架异常： 处理那些 Controller 之前就可能发生的异常（如请求参数绑定错误 MethodArgumentTypeMismatchException）。

📍 总结与最佳实践

  1. 抛出在 Service: 让 Service 层无负担地专注于业务逻辑，遇到问题直接抛出代表业务失败或技术故障的恰当异常。这是触发事务回滚的关键！
  2. 捕获转换在 Controller/Advice: 在 Controller 层（或全局 @ControllerAdvice）捕获这些异常。你的责任是隔离技术细节，将其翻译成用户/客户端能理解的、符合 API 契约的友好错误响应（包含正确的 HTTP 状态码和清晰的错误信息）。
  3. 全局处理器是优选： 为了统一处理和避免重复，优先使用全局的 @ControllerAdvice 处理通用异常和自定义异常的基类。Controller 内部的 @ExceptionHandler 主要用于处理该 Controller 特有的错误映射逻辑。
  4. 避免 Service 层”吃掉”异常或返回错误码： Service 方法内部不应自行捕获处理掉业务失败的异常或返回错误码/对象。这会使事务管理和错误传播失效。Service 应专注于报告失败（通过抛异常），由 Controller 负责最终的响应构建。
  5. 日志记录： 在 Service 层抛异常时记录（DEBUG 或 ERROR 级）。在全局异常处理器中也应记录捕获的异常（ERROR 级），这对于监控和问题排查至关重要。
分层 异常处理职责 关键原因 常用技术 示例
Service层 🛠️ 抛出异常（业务异常、包装后的技术异常） 保持事务完整性，清晰上报业务失败 throw new UserNotFoundException() 转账失败时抛出 InsufficientBalanceException
Controller层 🌐 捕获Service异常并转换用户友好响应 隔离技术细节，构建符合API契约的响应 try-catch 或 @ExceptionHandler 捕获服务层异常，返回400状态码+JSON错误详情
全局异常处理器 🧩 统一处理跨Controller的通用异常 避免重复代码，统一响应格式 @ControllerAdvice+@ExceptionHandler 集中处理所有参数验证失败异常

因此，异常处理的职责在分层架构中是协作完成的：Service层负责“发现问题并抛出”，Controller/全局处理器负责“翻译并告诉用户”。 📌
