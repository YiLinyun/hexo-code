## 一、摘要

本次测试希望回答三个问题：

1. `/sa` 接口的 QPS 为什么长期停留在 100～120 左右？
2. 高频日志到底只是消耗磁盘，还是会显著影响 CPU、响应时间和数据库连接池？
3. 把 Druid 最大连接数从 20 提高到 40，能否真正解决吞吐瓶颈？

最终结论如下：

- **同步日志是当前最主要的性能放大器。** 关闭控制台、文件和 MyBatis DEBUG 输出后，`/sa` 最高一轮平均 QPS 从约 111 提升到约 283；`profileSummary` 稳态平均 QPS 从约 322 提升到约 716。
- **连接池满载多数情况下是结果，不是根因。** 日志开启时，20 个和 40 个连接都可能长时间跑满；日志关闭后，`profileSummary` 在 716 QPS 下最大只观察到 16 个活跃连接。
- **MySQL 本身没有表现出慢查询或锁竞争。** `profileSummary` 的四条核心 SELECT 平均约 0.46～0.57ms；测试期间没有慢查询和行锁等待。
- **`profileSummary` 日志关闭后的主要剩余延迟来自远程 Redis。** 每请求一次 Redis GET，平均首响应约 385ms，P95约 694ms，P99约 1.25s。Redis 位于远程内网并通过 VPN 访问，这与尾延迟波动高度一致。
- **CPU、Heap 和 GC 不是当前首要瓶颈。** 日志关闭后吞吐翻倍，但 Java CPU 反而下降约 62%，GC开销始终低于1%。

这次测试也说明：性能问题不能只看“CPU有没有跑满”。大量同步日志、连接持有时间、网络往返和请求排队，都可能在 CPU 尚有余量时提前限制吞吐。

## 二、测试背景

最初本地应用需要通过 VPN/跳板网络访问远程 MySQL，接口响应时间受到网络环境影响。为了隔离数据库网络变量，本次测试先将应用主数据源切换到本机 MySQL 8.0 的同名测试库；Redis仍保持远程内网访问，因此后续结果中可以分别观察本地 MySQL 和远程 Redis 的影响。

测试按以下顺序进行：

1. 保持原始日志配置和 Druid 最大连接数20，测试 `/sa`。
2. 关闭日志，连接数保持20，再次测试 `/sa`。
3. 恢复日志，将连接数提高到40，第三次测试 `/sa`。
4. 保持日志开启和连接数40，测试读接口 `profileSummary`。
5. 关闭日志、连接数保持40，再次测试 `profileSummary`。

这种设计可以观察日志和连接池两个变量的影响。需要说明的是，各轮 `/sa` 的有效并发和升压轨迹并不完全相同，因此更适合判断趋势，不能把每个百分比都视为严格实验室条件下的因果值。`profileSummary` 两轮的 Tomcat 并发都接近600，A/B可比性更高。

## 四、指标口径与数据采集

主要指标来自 Spring Boot Actuator 和 Prometheus：

- QPS：HTTP请求计数器30秒速率；
- 平均耗时：HTTP耗时总和速率 ÷ 请求计数速率；
- P95/P99：HTTP直方图按1分钟窗口计算；
- Java/整机CPU：`process_cpu_usage`、`system_cpu_usage`；
- 数据库连接池：活跃连接、最大连接数；
- Tomcat：busy线程数；
- JVM：GC开销、Heap使用率；
- 日志：Logback事件计数，以及日志文件大小和最后写入时间；
- MySQL：Performance Schema语句摘要、慢查询、锁等待和连接状态；
- Redis：Lettuce命令首响应直方图。

需要特别注意：`logback_events_total` 反映代码“尝试记录日志”的次数。即使Logger或Appender被关闭，Micrometer过滤器仍可能统计日志调用，因此判断是否真正写盘还需要同时检查文件大小和最后写入时间。

P95/P99是滚动窗口估算值，升压、降压和服务重启附近会受到前一窗口样本影响。报告对明显的启动异常单独标注，并以稳定阶段作为主要结论依据。

## 五、`/sa` 写接口测试

### 5.1 原始基线：日志开启，连接池20

有效窗口：`16:08:53–16:19:03`

| 指标 | 结果 |
|---|---:|
| 平均QPS | 110.8 |
| 峰值QPS | 155.8 |
| 平均响应时间 | 2.394s |
| Java CPU平均值 | 45.5% |
| 整机 CPU平均值 | 84.5% |
| JDBC活跃连接平均值 | 18.57 / 20 |
| 连接池满载比例 | 77.4% |
| Tomcat busy平均/最大 | 267 / 501 |
| P95峰值 | 13.29s |
| P99峰值 | 15.97s |

表面现象是“数据库连接池20太小”，但同时观察到日志峰值接近9700次/秒，INFO日志文件累计达到约1.57GB。GC、Heap和执行器队列均正常，没有证据表明内存或GC是主因。

MySQL摘要显示该写链路每批请求包含用户写入/更新、拦截或行为日志写入，并产生约两次提交。COMMIT平均约10.82ms，日志表INSERT平均约11.16ms，其他SQL大多低于2ms。MySQL没有慢查询和行锁等待，因此更像是高频事务提交、同步日志和连接持有时间共同导致的吞吐限制。

### 5.2 关闭日志，连接池保持20

关闭内容包括：控制台Appender、INFO/ERROR文件Appender、业务包日志、Spring日志和MyBatis日志。Prometheus指标保留。

#### 第一轮

有效窗口：`17:20:50–17:26:10`

| 指标 | 结果 |
|---|---:|
| 请求量 | 51,601 |
| 平均/峰值QPS | 161.8 / 218.6 |
| 平均耗时 | 0.757s |
| P95平均/峰值 | 1.123s / 1.431s |
| P99平均/峰值 | 1.942s / 3.203s |
| Java CPU平均/峰值 | 7.5% / 17.2% |
| 整机 CPU平均/峰值 | 23.2% / 62.2% |
| 连接池满载比例 | 3.1% |

#### 第二轮高压阶段

有效窗口：`17:26:40–17:32:00`

| 指标 | 结果 |
|---|---:|
| 请求量 | 91,525 |
| 平均/峰值QPS | 283.5 / 375.2 |
| 平均耗时 | 0.878s |
| P95平均/峰值 | 1.363s / 2.254s |
| P99平均/峰值 | 2.220s / 4.620s |
| Java CPU平均/峰值 | 18.0% / 32.0% |
| 整机 CPU平均/峰值 | 35.1% / 62.9% |
| 连接池满载比例 | 38.5% |

相对原始基线，高压轮平均QPS提高约156%，平均耗时降低约63%。吞吐更高的同时，Java和整机CPU反而明显降低。日志文件最后写入时间保持不变，确认没有实际写盘。

这证明日志影响并非简单等价于“每秒写入多少MB”。真正的成本还包括：

- 日志级别判断和消息构造；
- 对象字符串化和参数格式化；
- Appender内部锁竞争；
- 大量小写入和系统调用；
- 控制台渲染；
- 在事务或Mapper调用期间延长连接持有时间。

### 5.3 恢复日志，连接池提高到40

主阶段：`17:42:25–17:47:40`

| 指标 | 主阶段 | 尾段 |
|---|---:|---:|
| 平均QPS | 129.3 | 102.8 |
| 峰值QPS | 203.5 | 128.2 |
| 平均耗时 | 1.094s | 2.398s |
| P95 | 1.503s | 3.164s |
| P99 | 1.731s | 3.630s |
| Java CPU平均值 | 33.1% | 33.4% |
| 整机 CPU平均值 | 69.4% | 70.6% |
| 连接池满载比例 | 53.1% | 77.3% |

连接池从20翻倍到40后，只在主阶段获得有限改善；持续压测后，连接池再次长时间满载，平均QPS回落到约103，平均耗时恢复到约2.4秒。

本轮INFO日志增长约1.21GB。日志事件平均约5700～7400次/秒，峰值超过11600次/秒。结果表明，增加连接数无法抵消同步日志和事务链路延长连接持有时间的问题；更多连接还会放大并发写入、提交和日志竞争。

### 5.4 `/sa` 结论

| 场景 | 平均QPS | 平均耗时 | 连接池满载 |
|---|---:|---:|---:|
| 日志开启 + 20连接 | 110.8 | 2.394s | 77.4% |
| 日志关闭 + 20连接（高压轮） | 283.5 | 0.878s | 38.5% |
| 日志开启 + 40连接（主阶段） | 129.3 | 1.094s | 53.1% |
| 日志开启 + 40连接（尾段） | 102.8 | 2.398s | 77.3% |

对 `/sa` 而言，日志开关比连接池从20扩到40的影响更大。连接池满载是事务执行和日志处理变慢后的排队表现，而不是唯一根因。

## 六、`profileSummary` 读接口测试

### 6.1 接口调用链

该接口每次请求大致执行：

1. 根据 `publicId` 查询用户；
2. 查询用户有效道具；
3. 从Redis读取邀请奖励弹窗；
4. 查询当天签到状态；
5. 查询用户拥有的勋章；
6. 组装结果并执行响应加解密。

即每请求约4次MySQL SELECT和1次远程Redis GET。MySQL Performance Schema数据显示，四条核心SELECT平均耗时约0.46～0.57ms，没有慢查询和锁等待，索引命中正常。

有效窗口：`17:57:25–18:02:45`

| 指标 | 结果 |
|---|---:|
| 请求量 | 102,768 |
| 平均/峰值QPS | 322.1 / 456.4 |
| 平均耗时 | 1.724s |
| P95平均/峰值 | 2.321s / 2.725s |
| P99平均/峰值 | 2.611s / 3.111s |
| Java CPU平均/峰值 | 36.6% / 48.2% |
| 整机 CPU平均/峰值 | 61.5% / 85.5% |
| JDBC活跃连接平均/最大 | 30.8 / 40 |
| 连接池满载比例 | 72.3% |
| Tomcat busy平均/最大 | 545 / 601 |

本轮INFO日志约增加558MB。在约600个并发请求线程下，40个连接有72.3%的采样时间全部占用。由于MySQL单条查询本身很快，连接池饱和更可能来自高频借还连接、每请求多次串行查询，以及MyBatis/业务日志延长Mapper调用和连接占用时间。

### 6.3 日志关闭，连接池保持40

Java在 `18:14:09` 重启。`18:16:25–18:16:35` 出现一段约10秒的升压异常：Tomcat busy为601，但JDBC和CPU均不高，判断为压测升压或遗留请求积压，不纳入稳态结论。

稳态窗口：`18:17:35–18:20:55`

| 指标 | 结果 |
|---|---:|
| 平均/峰值QPS | 716.5 / 981.2 |
| 平均耗时 | 0.848s |
| P95平均/峰值 | 1.588s / 2.913s |
| P99平均/峰值 | 2.890s / 4.960s |
| Java CPU平均/峰值 | 约14.0% / 25.8% |
| 整机 CPU平均/峰值 | 约22.1% / 36.9% |
| JDBC活跃连接平均值 | 小于1 |
| JDBC最大活跃连接 | 16 |
| 连接池满载比例 | 0% |
| Tomcat busy平均/最大 | 约560 / 601 |
| HTTP错误 | 0 |

与日志开启相比：

- 平均QPS提高约122%；
- 峰值QPS提高约115%；
- 平均耗时降低约51%；
- P95降低约32%；
- Java CPU降低约62%；
- 整机CPU降低约64%；
- 连接池从72.3%时间满载降为0%。

日志文件在有效压测期间没有发生写入。虽然日志调用计数仍然很高，但这些调用未进入控制台和文件Appender。

### 6.4 剩余尾延迟：远程 Redis

日志关闭后，数据库连接池已经不再构成瓶颈，但接口仍有明显P99波动。Lettuce指标显示，本轮每个请求对应一次远程Redis GET：

| Redis GET指标 | 结果 |
|---|---:|
| 平均QPS | 638.2 |
| 平均首响应 | 385ms |
| P95 | 694ms |
| P99 | 1.247s |
| P99峰值 | 2.566s |

Redis平均耗时约占接口稳态平均耗时的45%。结合Redis位于远程内网并依赖VPN访问，可以判断：**日志关闭后，远程Redis网络往返已成为 `profileSummary` 平均延迟和尾延迟的主要组成。**

### 6.5 `profileSummary` A/B结论

| 场景 | 平均QPS | 平均耗时 | P95 | 连接池满载 |
|---|---:|---:|---:|---:|
| 日志开启 + 40连接 | 322.1 | 1.724s | 2.321s | 72.3% |
| 日志关闭 + 40连接 | 716.5 | 0.848s | 1.588s | 0% |

两轮并发规模都接近600，因此该A/B结果具有较强可比性。日志关闭后吞吐翻倍、CPU下降、数据库连接池解除饱和，足以确认同步日志是首要瓶颈；Redis则是日志关闭后的下一优先级。

## 七、瓶颈证据链

### 7.1 第一优先级：同步日志

证据包括：

- 关闭日志后，两个接口吞吐都显著提升；
- 吞吐提高的同时CPU下降，而不是上升；
- 日志开启时连接池频繁满载，关闭后连接占用大幅下降；
- `/sa` 日志开启轮数分钟写入约1.21GB；
- `profileSummary` 日志开启轮约写入558MB；
- 日志关闭后文件时间戳保持不变；
- MySQL查询、锁、GC和Heap均没有对应的饱和证据。

这说明同步日志不仅消耗磁盘吞吐，还会通过消息构造、对象序列化、锁、系统调用、控制台渲染和事务内执行时间影响整个请求链路。

### 7.2 第二优先级：远程 Redis/VPN

`profileSummary` 的远程Redis GET平均385ms、P99约1.25s。对于一个本地MySQL查询均低于1ms的读接口，这个网络往返已经非常突出。Redis远程访问解释了日志关闭后平均耗时仍接近0.85秒，以及P99仍有2～5秒波动。

### 7.3 第三优先级：写接口事务和提交次数

`/sa` 是写密集链路，约两次提交/请求，COMMIT和日志表INSERT平均约10～11ms。单次不算慢，但在数百QPS下会形成大量同步提交和刷盘。日志如果位于事务范围内，还会进一步延长连接持有时间。

### 7.4 连接池：需要调优，但不是越大越好

连接数从20提高到40并未在日志开启时持续提升 `/sa` 吞吐；`profileSummary` 日志关闭后最大只使用16个连接。这说明继续扩大连接池缺乏依据，还可能增加MySQL并发、上下文切换和磁盘提交压力。

当前合理区间可暂定20～40，最终值应在日志优化、Redis隔离后重新压测决定。

## 八、优化建议

### P0：立即处理日志配置

1. 生产环境关闭MyBatis DEBUG，至少调整到WARN。
2. 高频接口不记录完整参数、完整用户对象、SQL参数和结果集。
3. 压测和生产环境关闭同步控制台输出。
4. 必须保留的文件日志使用异步Appender，并设置队列上限和丢弃策略。
5. 对成功请求日志采用采样或聚合，例如每1000次输出一次统计。
6. 错误日志保留必要上下文，但避免在热点路径重复输出同一异常。
7. 避免在事务内部执行非必要日志格式化和对象字符串化。

日志优化的目标不应该只是“少写几个GB”，而是降低每请求日志数量、缩短连接持有时间，并避免请求线程同步等待Appender。

### P1：处理 `profileSummary` 的远程 Redis

1. 本地基准测试使用本地Redis，先获得不包含VPN的服务上限。
2. 生产部署确保应用与Redis处于低延迟网络区域。
3. 为邀请奖励弹窗增加短TTL本地缓存或空值缓存，避免每请求远程GET。
4. 设置明确的Redis连接、命令超时和降级逻辑。
5. 如果该字段不是页面首屏强依赖，可以拆分为异步加载。
6. 单独监控Redis GET的平均值、P95、P99、超时率和连接池状态。

### P1：缩短 `/sa` 写事务

1. 检查两次提交是否可以合并为一次事务。
2. 将非核心行为日志、拦截记录等改为异步事件或消息队列。
3. 缩小 `@Transactional` 范围，不在事务中执行无关日志和网络调用。
4. 对可批量写入的数据采用批处理。
5. 保持幂等和唯一键约束，避免为了性能牺牲一致性。

### P2：优化 `profileSummary` 查询结构

该接口四条SQL均很快，优先级低于日志和Redis，但仍可降低连接借还次数：

1. `SELECT *` 改为只查询接口需要的字段。
2. 评估将签到、道具和勋章信息聚合为更少的查询。
3. 对稳定数据或空结果使用短TTL缓存。
4. 如果需要一致的用户视图，可在只读事务内复用同一连接，但要避免为了减少借还而扩大事务范围。

### P2：连接池和线程池

1. 不建议直接把连接池继续扩大到80或更高。
2. 连接池大小应结合数据库CPU、磁盘、单查询耗时和连接持有时间决定。
3. Tomcat线程数应与下游容量匹配，必要时设置限流和背压，避免数百线程同时等待少量连接。
4. 对获取数据库连接的等待时间增加独立监控。

为进一步获得可复现、可用于容量规划的数据，建议使用完全相同的请求数据、并发曲线和测试时长完成以下矩阵：

| `/sa` 日志关闭 + 20连接 vs 40连接 | 单独验证连接池大小影响；本次缺少“关闭日志+40连接”组 |

## 十、测试限制

1. 服务、MySQL、压测客户端和部分监控组件位于同一台开发机，整机CPU包含多个进程。
2. `/sa` 各轮有效并发和升压轨迹并不完全一致，适合判断趋势，不适合做精确容量承诺。
3. Redis仍通过VPN访问，`profileSummary` 结果不能代表应用与Redis同机房时的性能。
4. P95/P99来自Prometheus滚动直方图，窗口边界附近会带入前一阶段样本。
5. 日志文件增长量包含少量启动和后台日志，应视为近似值。
6. 本次没有完成 `/sa`“日志关闭 + 连接池40”组，因此尚不能确定写接口的最佳连接数。

## 十一、最终结论

本次压测最重要的发现不是“连接池应该设成多少”，而是：**同步日志改变了整个请求执行模型。** 它增加CPU和锁竞争、制造大量小写入、延长数据库连接持有时间，最终表现为连接池满载、Tomcat线程堆积、QPS下降和尾延迟上升。单纯扩大连接池只能短暂缓解排队，不能消除根因。

对当前系统，建议按照以下顺序推进：

1. 先关闭或异步化热点路径日志，尤其是MyBatis DEBUG和每请求成功日志；
2. 将本地压测的Redis也切到本机，隔离VPN变量；
3. 优化 `profileSummary` 的远程Redis访问和空值缓存；
4. 缩短 `/sa` 事务、减少提交次数、异步化非核心写入；
5. 最后再通过严格A/B测试确定连接池大小，而不是继续盲目扩容。

在本次硬件和约600并发条件下，`profileSummary` 日志关闭后的稳态平均QPS约716、峰值约981，且CPU、GC、Heap和数据库连接均有余量。这意味着系统仍有继续提升空间；下一阶段真正需要解决的是远程Redis尾延迟和写接口事务结构，而不是继续增加数据库连接。
